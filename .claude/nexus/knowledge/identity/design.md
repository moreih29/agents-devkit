# Design

## 역할 구조

### Lead (메인)

의도 발굴(Discovery) + 조율 + 사용자 소통. 에이전트 위임이 기본값.

**기본 동작**: 최소 Director 스폰. 이 비용은 넥서스를 선택한 이상 감당하는 비용.

**Lead 직접 실행 조건** (모두 충족 시만):
1. 사용자가 정확한 변경 지시를 했다
2. 단일 파일 수정으로 완결된다
3. 코드 구조 이해가 불필요하다 (오타, 린트 에러, 상수 변경 등)

**판단 참조**: codebase knowledge, memory, 요청 텍스트 복잡도 신호. 코드 직접 읽기/LSP/웹 검색은 금지 (분석은 에이전트의 몫).

**판단 원칙**: 의심스러우면 높은 레벨로. 과소 판단 비용 >> 과대 판단 비용.

### 에이전트 (6개)

| 역할 | 기능 | 카테고리 |
|------|------|----------|
| **Director** | 의도 대변(Advocacy) + Why/What + task 소유 | Decide |
| **Architect** | 기술적 실현 가능성, 코드 설계 | How |
| **Postdoc** | 방법론, 증거 평가, synthesis | How |
| **Engineer** | 코드 구현, 디버깅 | Do |
| **Researcher** | 웹 조사, 실험 | Do |
| **QA** | 검증, 테스트, 보안 리뷰 | Check |

**핵심 구분**: 의도 발굴(Lead) vs 의도 대변(Director). Lead는 사용자와 직접 대화하여 의도를 발굴하고, Director는 팀 내부에서 그 의도를 지키는 수호자.

Architect/Postdoc은 분리 유지 — 관점의 다양화가 품질에 기여한다. 팀 경계 없이 목표에 따라 자유 조합.

## 태그 체계

| 태그 | 모드 | 설명 |
|------|------|------|
| `[consult]` | 상담 | 의도 발굴. task 파이프라인 없이 대화 |
| `[do]` | 실행 | task 파이프라인 + 에이전트 위임 + Progressive Depth + Dynamic Composition |
| `[d]` | 기록 | 결정 기록 |

"풀팀 동원" 같은 오버라이드는 자연어로 Lead에게 전달 (User Sovereignty).

## 정보 관리 체계

```
.claude/nexus/
├── core/            ← 정보 (넥서스 관리)
│   ├── identity/    ← 철학, 미션, 설계 원칙, 로드맵
│   ├── codebase/    ← 코드 구조, 아키텍처, 도구 (nx-sync 자동 갱신)
│   ├── reference/   ← 외부 조사 결과, 경쟁사 분석, 기술 비교 (에이전트/사용자 수동)
│   └── memory/      ← 과거 교훈, 실패 패턴, 프로젝트 수준 학습 (task_close 시 자동 추출)
├── rules/           ← 지시 (넥서스 관리, 도메인별 커스텀)
└── config.json
```

| 계층 | 갱신 주체 | source of truth |
|------|----------|----------------|
| identity | 넥서스가 사용자에게 물어서 | 사용자 |
| codebase | nx-sync 자동 | 프로젝트 코드 |
| reference | 에이전트/사용자 수동 | 외부 세계 |
| memory | task_close/루프 감지 시 자동 | 과거 경험 |

**core/ 소유권**: 넥서스가 관리. 생성(사용자에게 물어서), 갱신(변경 감지 시 자동), 삭제(넥서스 판단).

**rules/ 소유권**: 넥서스가 관리. 사용자가 자연어로 요청하면 Lead가 판단하여 구조화 저장. 도메인/작업 유형별 자유 명명(blog.md, api.md 등). 사용자는 rules 폴더의 존재를 알 필요 없음 — Lead가 관련 작업 시 태그 매칭으로 자동 주입.

**identity 필수 파일**: mission.md(존재 이유 + 핵심 가치 + 행동 원칙), design.md(핵심 설계 결정), roadmap.md(Phase별 달성 목표). 프로젝트 규모와 무관하게 필수. 최소 구조 가이드(필수 섹션)를 따르되 내용/분량은 자유.

Memory 기준: "이 정보가 없으면 같은 실수를 반복할 것인가?" — 실수 방지 + 자기 발전 메커니즘.

## 컨텍스트 엔지니어링

에이전트 스폰 시 briefing은 하이브리드 방식: 시스템이 역할별로 4계층에서 필요한 정보를 자동 수집 + Lead가 의도/맥락을 추가.

| 역할 | identity | codebase | reference | memory |
|------|----------|----------|-----------|--------|
| Director | 전체 | 요약 | 관련 항목 | 전체 |
| Architect | 원칙 | 전체 | 기술 관련 | 기술 교훈 |
| Postdoc | 원칙 | 관련 | 전체 | 방법론 교훈 |
| Engineer | — | 관련 모듈 | — | 해당 영역 |
| Researcher | 미션 | — | 전체 | 조사 교훈 |
| QA | 원칙 | 관련 모듈 | — | 실패 패턴 |

## 하네스 메커니즘

### 2단계 검증 (Two-layer Verification)

- **Director**: 의도 검증 (항상 — 기본 스폰이므로)
- **QA**: 산출물 검증 (Director 재량 + 자동 스폰 조건)

QA 자동 스폰 조건 (하나라도 해당 시):
- 변경 파일 3개 이상
- 기존 테스트 파일 수정
- 외부 API/DB 접근 코드 변경
- memory에 해당 영역 실패 이력 존재

### 루프/실패 감지 + 단계적 에스컬레이션

감지: 동일 에러 반복, 같은 파일 3회 이상 수정, 동일 검색 반복.

에스컬레이션 체인:
1. 해당 에이전트 중단 → Director 보고
2. Director가 대안 결정 (다른 에이전트 재배정, Architect 자문 등)
3. Director도 해결 못함 → Lead가 사용자에게 알림

최종 판단은 사용자 (User Sovereignty).

### Memory 자동 기록

- 루프 감지 시: "X 방식은 Y 이유로 실패" 자동 기록
- task_close 시: 사이클 교훈 자동 추출 (history.json → memory/)

## 런타임 상태

consult.json, decisions.json, tasks.json 구조는 유효하나, 형태의 자유도를 유지한다 — 더 좋은 방법이 있으면 변경 가능.

## Consult 원칙

1. **적극적 의도 발굴** — 사용자가 명확히 하지 못하는 것을 적극적으로 찾아라. 말 이면의 근본적 목표를 인터뷰로 발굴.
2. **선제적 탐색 확장** — 코드/knowledge뿐 아니라 외부 자료(웹 검색, 기술 조사)도 선제적 수행. 에이전트 소환으로 정보 수집 가능.
3. **가설 기반 질문** — 빈 질문이 아닌 탐색 결과에 근거한 가설을 세우고 사용자에게 확인.
4. **Progressive Depth** — 요청 복잡도에 따라 상담 깊이 자동 조절.
5. **객관적 반박** — 사용자 의견에 동의만 하지 마라. 반론 근거가 있으면 적극 반박. 넥서스는 예스맨이 아니다.
