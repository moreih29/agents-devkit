# ADR: 훅 격리 전략

## 상태
확정 — Phase별 전략 적용 (2026-03-19)

## 맥락
omc는 모든 훅을 별도 프로세스로 실행하여 안정적이지만 느리다 (이중 프로세스 스폰, ~100ms/호출).
omo는 모든 것을 인프로세스로 실행하여 빠르지만 장애가 전파된다.
Nexus는 Claude Code 플러그인이므로, PreToolUse/PostToolUse 훅은 반드시 hooks.json 별도 프로세스여야 한다.

## 결정
Phase별로 최적화 전략을 적용한다:

### Phase 1: 경량 스크립트 (기본안)
- 모든 훅을 단일 CJS 스크립트로 실행 (omc의 이중 스폰 제거)
- 예상 오버헤드: ~30-50ms/호출

### Phase 2: 선택적 등록
- 기본: Gate + Tracker만 등록 (평소 오버헤드 0)
- 모드 활성 시: Pulse 동적 추가
- hooks.json 런타임 수정 가능 여부 확인 필요

### Phase 3: 상주 데몬 (필요 시)
- SessionStart에서 데몬 실행
- 훅 스크립트가 Unix socket으로 데몬에 질의 (~5ms)
- 데몬이 인메모리 상태 유지

## 근거
- Phase 1부터 동작하는 것을 만든 후, 실제 오버헤드 측정 결과에 따라 최적화
- 설계는 훅 모듈의 인터페이스만 고정하고, 실행 방식은 교체 가능하게 함
- MCP 서버 인프로세스 처리는 Claude Code 아키텍처상 불가능 확인됨

## 대안
1. **MCP 인프로세스** → Claude Code에서 기술적으로 불가. 기각.
2. **처음부터 데몬** → 구현 복잡도가 MVP에 부적합. 기각.
3. **모든 훅 항상 활성** → 불필요한 오버헤드. 기각.
