# ADR: 이중 저장소 아키텍처 결정

## 상태
확정 (2026-03-19)

## 맥락
에이전트 오케스트레이션 시스템에서 상태/지식/계획을 어디에 저장할 것인가.

### 기존 시스템의 문제
- **omc**: 모든 것을 `.omc/` (gitignore)에 저장 → 장기 지식이 세션/머신 변경 시 소실
- **omo**: 인메모리 상태 → 세션 종료 시 모든 것 소실

### 핵심 통찰
에이전트 오케스트레이션의 본질은 "최적화된 컨텍스트를 역할에 맞는 에이전트에 주입하는 것"이다. 프로젝트 지식과 계획은 모든 세션, 모든 팀원이 접근할 수 있어야 에이전트가 올바른 컨텍스트를 받을 수 있다.

## 결정
저장소를 두 개로 분리한다:
1. `.claude/nexus/` — git 추적 (장기 지식, 계획, 결정 근거)
2. `.nexus/` — .gitignore (세션 상태, 단기 메모, 로그, 캐시)

## 근거
- 장기 지식은 코드와 동일한 수준의 자산이므로 git으로 버전 관리
- 세션 상태는 다른 세션에 무관하므로 휘발성이 적절
- plans는 브랜치별 파일(`plans/{branch}.md`)로 충돌 방지, 병합 시 삭제
- knowledge 파일은 PR 리뷰 대상 (AI가 기록한 것도 사람이 검증)

## 대안
1. **모두 gitignore** (omc 방식) → 장기 지식 소실 위험. 기각.
2. **모두 git 추적** → 세션 상태가 git을 오염. 기각.
3. **외부 DB** → 복잡도 과잉, 로컬 개발에 부적합. 기각.
